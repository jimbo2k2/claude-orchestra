Orchestra v2

Three-tier autonomous development with DDD governance

Last reviewed: Not yet reviewed Version: v2 design draft Date: 2026-03-25

What is Orchestra v2?

Orchestra v2 replaces the v1 "human writes plan, Claude executes checklist" model with a three-tier autonomous development system. The governance layer uses numbered, archivable protocols (T/D/C-numbers). The execution layer includes a built-in codewriting loop for React Native + Expo + Supabase with mandatory code review, UI testing, and debug passes.

What stays from v1

  • Session loop with crash recovery
  • Hook system (stage, verify, commit)
  • Model selection (sonnet default, opus on recommendation)
  • HANDOVER.md and INBOX.md
  • orchestra init, orchestra run, orchestra status, orchestra reset

What is removed

  • orchestra graduate — continuous DDD documentation replaces end-of-build consolidation
  • Checkbox-format TODO, append-only DECISIONS/CHANGELOG in .orchestra/
  • Read-only PLAN.md
  • Generic docs/ scaffolding

What is new

  • Three-tier planning (Strategic → Tactical → Tertiary) with codewriting loop execution
  • Numbered, archivable CHANGELOG protocol (C-numbers)
  • Codewriting inner loop (Code → Review → 2nd pass → UI Test → Debug)
  • Two-layer acceptance criteria (Standing AC + Task AC)
  • Decomposition review + plan coherence check (quality gates)
  • Task dependency tracking (Depends: field) with blocked-task skip
  • Separate toolchain.md for stack-specific commands
  • Pre-flight validation at orchestra run
  • .orchestra/config maps governance to project-specific paths

.orchestra/ Directory

.orchestra/ ├── config # Governance paths, plan, toolchain, standing AC ├── toolchain.md # Stack-specific build/test/capture commands ├── standing-ac.md # Standing acceptance criteria (human-authored) ├── HANDOVER.md # Session-to-session context (overwritten each session) ├── INBOX.md # Async human → Claude messages ├── COMMIT_MSG # Temporary commit message from session ├── session-logs/ # Raw session output └── archive/ # Archived handovers + logs after reset Governance files live in the PROJECT, not here: TODO/TODO.md, Decisions/DECISIONS.md, Changelog/CHANGELOG.md

Task statuses

  • OPEN: created and ready for pickup (default for new tasks)
  • IN_PROGRESS: a session has started work
  • COMPLETE: work finished and verified
  • BLOCKED: cannot proceed without human input
  • PROPOSED: human-gated; created by decomposition when scope expansion detected

Task eligibility rules

  • Eligible: status is OPEN; all Depends: tasks are COMPLETE
  • PROPOSED: skipped until human changes status to OPEN
  • Blocked skip: if a task depends on a BLOCKED task, it is ineligible but not itself BLOCKED — the session moves past it
  • Parent completion: strategic tasks are marked COMPLETE automatically when all children are COMPLETE
  • Circular deps: caught during decomposition review

Archiving

When the active file exceeds its threshold (configurable per artifact), a session archives completed entries to an immutable batch file, keeping one-line summary rows in the index.

Three-Tier Planning

Tier 1 — Strategic (Human + Claude)

Interactive session before orchestra run. Produces a plan file and coarse T-numbered tasks. Feature / bounded context scope. Human directs, Claude assists.

Example: T060: Build Journal post creation screen with media upload

Tier 2 — Tactical (Claude, autonomous)

Session decomposes a strategic task into tactical tasks with Parent and Depends links. Screen / component scope. Validated by decomposition review + plan coherence check before execution.

Example: T064: Create PostCreation screen scaffold (Parent: T060)

Tier 3 — Tertiary (Claude, autonomous, rare)

Further decomposition when a tactical task is more complex than expected. Same pattern, one level deeper. Maximum depth. If a fourth level is needed → BLOCKED.

Example: T069: Camera permission + capture (Parent: T065)

Quality gates (after tier 2/3 decomposition)

  1. Decomposition review (internal): sufficient coverage, correct ordering, no overlap, no circular deps
  2. Plan coherence check (external): aligns with strategic goals, bounded context docs, PUBLISHED-LANGUAGE.md, cross-context dependencies, project phase

Outcomes: both pass → execute. Scope expansion → flag INBOX, continue with safe subset. Fundamentally incoherent → BLOCKED.

Task dependency tracking

  • Parent: — decomposition hierarchy ("what strategic task does this serve?")
  • Depends: — execution ordering ("what must complete first?")
  • Dependencies can cross parent boundaries
  • A BLOCKED task automatically makes its dependents ineligible (but not BLOCKED themselves)

Codewriting Inner Loop

Runs for each tactical/tertiary task that involves code changes. Hardcoded for React Native + Expo + Supabase via toolchain.md.

Loop steps

  1. Generate Task AC — write task-level acceptance criteria as children under Standing AC categories. Record in the task detail before coding starts.
  2. Write code — read toolchain.md for conventions and commands. Implement the component/hook/screen/query.
  3. Code review (mandatory) — run code-review skill pass. Produces issue list with severity ratings.
  4. Conditional 2nd pass — if issues found: fix all, re-run review. If clean: continue.
  5. UI test — Expo web + Puppeteer at target viewport. Two feedback channels (visual + structural) plus Supabase data verification.
  6. Debug pass — if failures: diagnose (layout vs logic vs state), fix root cause, loop back to UI test. If all AC pass: mark complete, write C-entry.

Two-layer acceptance criteria

Standing AC (.orchestra/standing-ac.md)

Permanent, human-authored. Applies to every UI task. Defines the categories:

  • Visual & Layout — rendering, viewport, regressions
  • Functional — interactivity, navigation, state handling
  • Data — Supabase persistence, RLS, optimistic updates
  • Code Quality — console errors, TypeScript, accessibility
  • Integration — navigation structure, no regressions

Task AC (generated per task)

Claude generates specific criteria under Standing AC categories before implementation. This prevents blind spots where a standing category has no task-level children.

### Task AC (T064: PostCreation screen scaffold)
Under "Visual & Layout":
  - Screen renders with header, back nav, save button
  - Form fields visible and tappable
Under "Functional":
  - Back button navigates to post list
  - Save disabled when title empty
Under "Data":
  - No database interaction at this stage

Commands

orchestra init [dir]

Scans for existing governance, inherits or creates. Creates .orchestra/ with config, toolchain template, standing AC template, HANDOVER, INBOX, session-logs.

  • Found + matches: inherit, write path to config
  • Not found: create full structure (directory, main file, CLAUDE.md protocol, archive/)
  • Found + conflicts: warn, pause for input (adopt / point / skip)

orchestra run

Pre-flight validation, then session loop until COMPLETE, BLOCKED, or limits reached.

Pre-flight checks:

  • PLAN_FILE exists and non-empty
  • TOOLCHAIN_FILE exists and non-empty
  • STANDING_AC_FILE exists and non-empty
  • Governance files at config paths exist
  • Eligible tasks in TODO

orchestra status

Reads from config-mapped governance files. Shows task progress, decision/changelog counts, session count, plan objective, last handover.

orchestra reset [--label]

Archives HANDOVER + session-logs. Resets HANDOVER, clears INBOX processed messages. Does not touch governance files.

Hooks

verify-completion.sh Stop Verify T-numbered task progressed, C-entry added, D-entries if decisions made. Uses Haiku. Reads paths from config.

Crash Recovery

  1. Compare governance file checksums to pre-session snapshot
  2. If governance files changed: partial state → recovery commit → next session uses recovery prompt (governance may be inconsistent)
  3. If governance unchanged but code files modified → recovery commit → next session uses recovery prompt
  4. If nothing changed: session died before acting → next session uses normal prompt
  5. Consecutive crash counter increments (resets on success). Max reached → exit 1.
